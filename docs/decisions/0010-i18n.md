# 0010 — 多语言支持(i18n):English Default,中英文双轨

> [English](./0010-i18n.en.md) | **中文**

- Status: Accepted
- Date: 2026-05-01

## Context

knowlet 之前的所有用户面字符串(CLI 提示、面板标题、Web 按钮、profile 模板)散落在代码各处,而且默认是中文 / 英文混在一起(注释多为英文,profile 模板是中文,banner 文本是英文)。

随着接入外部用户(scratch-your-own-itch 阶段后)临近,这种"开发者母语决定 UI 语言"的模式不再可持续:

- 母语英语的用户开 vault 进 chat,会看到 banner 是英文但 profile 模板是中文,体验割裂。
- 即使是中文用户,如果以后跟英语贡献者协作,共享配置 / 任务 / 草稿目录会发生 surprise。
- 文档已经按 [feedback_bilingual_docs](../../) 的要求做中英双轨,代码层却没有对齐。

用户 2026-05-01 明确指示:**"knowlet 需要支持多语言,English Default。现阶段需要支持中英文。"**

需要把 i18n 落到代码层,并把"OSS 默认"明确为 English。

## Decision

### 1. 默认语言:**English**

- 新 vault 默认 `config.general.language = "en"`。
- profile 默认模板默认英文(`DEFAULT_PROFILE_TEMPLATE_EN`),中文版作为 alternate。
- 所有 UI 字符串先有英文版,中文版是翻译。
- 文档(README、ADR、design docs)继续按 bilingual-docs memory 的"中文初稿 → commit 前补英文"流程,但**初稿可以从英文也可以从中文起手**(初稿语言不绑定);commit 前两版都到位即可。

### 2. 实现:**手写最小 catalog,不引入 gettext / babel**

- `knowlet/core/i18n.py` 一个文件,内部两个 dict(`_EN`、`_ZH`)。
- 公开 API 只有四个:`set_language(lang)`、`current_language()`、`init_from_env()`、`t(key, **vars)`,以及 `all_keys(lang)` 给 web 拉 catalog 用。
- 缺失 key fallback 到英文;英文也缺则**返回 key 本身**,让漏翻在开发期立刻可见。
- 当前规模约 80 个 key,gettext 工具链(`xgettext` / `.po` / `msgfmt`)的开销不值。需要时(>500 key)再升级,迁移成本可控。

**为什么不用 gettext:** 引入构建步骤(.po → .mo)、agent 难以快速增删 key、`pybabel extract` 对 f-string / Rich markup 的支持不稳。手写 dict 不增加构建依赖,任何 agent 都能直接编辑 catalog。

### 3. 语言来源(运行时):**`config.general.language`,KNOWLET_LANG 可强制 override**

- 主来源:`<vault>/.knowlet/config.toml` 里 `[general] language = "en"`(新增 section)。
- Override:env var `KNOWLET_LANG` 在 dev / 测试场景手动切;但日常使用不依赖。
- **不**自动从 `Accept-Language` HTTP header / locale 读 —— 单用户本地工具,没必要;且会带来"为什么我打开就变成另一种语言"的 surprise。
- CLI 在每个命令通过 `_load_config_or_default` 流程里调 `set_language(cfg.general.language)` 激活。
- Web FastAPI 在 lifespan 启动时调一次,`/api/health` 返回 `language` 让前端能显示当前状态;前端启动时拉 `/api/i18n/{lang}` 把 catalog 拉到 `window.I18N` 对象,渲染时遍历 `[data-i18n]` 属性替换 textContent / placeholder。

### 4. **chat 内容语言**(LLM 回复)与 UI 语言**解耦**

- chat 系统提示里继续保留 "Reply in the same language the user used" —— LLM 自动镜像用户输入语言,**不**强制按 `config.general.language`。
- 这意味着英文 UI 用户问中文,模型也会用中文回。这是 feature(双语用户的真实场景),不是 bug。
- profile 注入到 system prompt 时不翻译 —— 用户怎么写就怎么注入,模型自己理解。
- 草稿提取 prompt 也保持原样("write in the source's main language") —— 来源是英文,草稿就是英文。

### 5. **CLI 入口字符串**:批量替换 `print(...)` → `console.print(t("..."))`

- 优先 migrate 高曝光路径:vault init banner、config init wizard panels、chat REPL 启动 banner、slash help body、_ensure_ready_or_wizard 的 onboarding 文案。
- 低曝光的 error message(异常路径里的 "task not found:" 等)**不一次性 migrate**,逐步沿用即可 —— 等到某个语言用户实际遇到再翻译。这是有意识地控制初始 i18n 工作量,避免"为了 100% 覆盖率燃烧 1 整天却交付不到核心价值"。
- Migration 入口 = `t(key, **kwargs)`,`{root}` / `{model}` 之类的占位符通过 `str.format` 注入。

### 6. **Web 入口字符串**:`data-i18n` 属性 + JS apply

- HTML 里给所有静态可见文字加 `data-i18n="key"`(textContent)或 `data-i18n-placeholder="key"`(input placeholder)。
- 动态 toast / 状态文本通过 JS 端的 `tt(key)` 函数取(`window.I18N` 是从 `/api/i18n/{lang}` 拉来的 dict)。
- 加载顺序:health → 拿 `language` → `loadI18n(lang)` → `applyI18n()` 替换 → 渲染数据。

### 7. **chat / sediment / mining extractor 的 prompt 不进 catalog**

- 这些 prompt 是给 **LLM 看的**,不是给用户看的;`t()` 是用户层翻译机制,不该混进来
- 翻译它们没价值 —— 用户的 UI 语言不影响 prompt 工程的有效性;double-translation(英文 → 中文 → LLM 内部又翻回去)只增加 token 不增加质量
- 留在 `chat/prompts.py` 作为英文常量
- **proxy 兼容性不在本 ADR 范围内**:某些 OpenAI-compat proxy(如经 Claude Code 路由)不严格遵循 `role: "system"`,这一层的 mitigation(把 system 指令折回 user message)写在**用到它的具体 prompt 文件里**(见 `chat/sediment.py:38-45`),而不是 i18n 层。本节早期版本曾错误暗示"ADR-0008 已经解决了这层"—— 不准确,2026-05-02 修订

## Consequences

### 好处

- **OSS-friendly default**:外部用户落地默认英文体验,不需要懂中文。
- **架构钉死**:任何新增字符串都进 `core/i18n.py`,既保证翻译完整性,又有"漏翻在开发期立刻可见"的反馈机制(返回 key 本身)。
- **不引入构建步骤**:agent 改字符串就是改 dict,不需要 `pybabel`。
- **Backward-compat 自然成立**:旧 config.toml 没有 `[general]` section → Pydantic 默认补 `"en"`;旧 vault 已有的 profile 文件**不**触动(`ensure_profile` 只在文件不存在时写默认模板)。
- **测试友好**:`set_language("zh")` 切换后 t() 可断言;`all_keys("zh")` 可对比 `all_keys("en")` 检查覆盖率。
- **chat 体验不退化**:UI 语言与回复语言解耦,bilingual 用户继续 happy path。

### 代价 / 约束

- **首批 catalog 不全 100%**。低曝光的 error message 还是裸字符串,这是有意为之;会留 lint-able TODO 在代码注释里。
- **HTML/JS 双方需要保持 catalog key 同步**。后端加 key 而前端 HTML 没补 `data-i18n`,前端就显示原 fallback 文本。需要在 review 时盯一下。
- **加新语言的成本**:加一份 `_LANG_CODE` dict + 一个 `SUPPORTED_LANGUAGES` 元组项;约 50 行翻译。可接受。
- **profile 模板第一次创建后不会自动重渲染**:语言切换之后,旧的 `users/me.md` 还是创建时的语言。这符合"用户拥有 vault 内容"的原则 —— knowlet 不主动覆盖用户文件。需要的话用户自己 `rm users/me.md && knowlet user edit` 重生成。
- **chat REPL prompt 是 `[bold cyan]you[/bold cyan]`(英文)还是 `[bold cyan]你[/bold cyan]`(中文)**根据 i18n 切换。这点 trivial 但要在测试里别 hardcode 字面值。

### 未来可能的扩展

- 加更多语言(ja / ko / es / fr 等):约 50 行翻译/语言。
- 真正的 RTL 支持(ar / he):需要给 CSS 加 logical properties。M5+ 再说。
- 用户在 chat 里说 "use English from now on" → 模型遵守:**已经成立**(系统 prompt 指令模型镜像用户语言),不需要新机制。
- 如果未来加 Web settings 面板,把 language 切换做成下拉 + 立刻 reload `/api/i18n/{lang}` —— 这是 5 行 JS 的事,不影响 ADR。

### 与现有 ADR 的关系

- **遵循** [ADR-0008](./0008-cli-parity-discipline.md):catalog 是 backend 单一事实来源,CLI / Web / 未来 desktop shell 都从同一份拉;不允许某层"自己内嵌一份字符串"。
- **遵循** [feedback_bilingual_docs](../../) memory:文档继续中英双轨;ADR 同步两版。
- **不冲突** [ADR-0002](./0002-core-principles.md):语言切换不影响数据格式;Note / Card / Draft / Task 的 frontmatter 字段名是英文(架构约定),用户写的内容仍是用户拥有。
- **不冲突** [ADR-0005](./0005-llm-integration-strategy.md):LLM 调用不变;system prompt 字符串保持英文。
