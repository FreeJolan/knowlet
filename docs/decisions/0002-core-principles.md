# 0002 — 三条核心原则

> [English](./0002-core-principles.en.md) | **中文**

- Status: Accepted
- Date: 2026-04-30

## Context

PKM 类项目极易陷入两个陷阱:一是过度依赖某个 LLM/云服务,失去离线可用性;二是数据被锁在自有格式或自有云中,用户迁移成本高。同时,随着特性扩张,如果没有边界约束,核心代码会很快变成"大泥球",任何一个新模块都需要改动核心。

为避免上述问题,需要在项目早期就把架构与产品上的红线确立下来,作为后续所有 ADR 与设计决策的底线。

## Decision

确立三条**不可动摇**的底层原则,任何特性、模块、外部依赖的引入都必须先过这三条:

### 1. AI 是可选增强,不是必需品

- 没有 AI(无网、无 API key、未配置模型)时,Knowlet 仍是一个**可用的笔记库**
- AI 能力以"加分项"形式暴露,而非"前置依赖"
- 核心数据结构、核心交互(写入、检索、复习)在零 AI 配置下必须能跑

### 2. 数据主权在用户

- 所有数据**本地优先**,以纯文本 / Markdown / Frontmatter 落盘
- 云端同步是**可选**(Git / WebDAV 等开放协议),不绑定任何专有云
- 用户随时可以打包带走全部数据,且离开 Knowlet 后这些数据在其他工具中仍可读

### 3. 能力插件化

- "定时拉取报告"、"OCR 卡片"、"问答考核"等能力做成**独立模块**
- 模块之间通过明确接口通信,互不直接依赖
- 用户/社区可以替换、新增、禁用任意能力模块,而不影响核心运行

## Consequences

**好处**

- 离线可用性 → 用户可在任何环境(飞机、地铁、内网)继续使用
- 数据可迁移 → 降低用户尝试成本,提升信任
- 模块边界清晰 → 社区贡献者更容易切入,也避免后续特性互相冲突("学习型要 SRS、情报型要订阅"等架构内耗)

**代价 / 约束**

- 任何"必须联网"或"必须用某 LLM"的优雅设计都要被拒绝,需多花精力做降级路径
- 数据格式必须保持开放,不能为了某个高级特性引入私有二进制格式
- 模块边界要严格守住,前期会比"一把梭"开发慢一些,但避免后期重构成本

**与其他 ADR 的关系**

- 与 [0001 — Wedge 破局点](./0001-wedge-learning-first.md) 协同:破局点聚焦学习型,但底层三原则保证未来扩张时不会被早期路径锁死
