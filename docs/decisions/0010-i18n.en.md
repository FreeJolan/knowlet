# 0010 — Multi-language support (i18n): English default, EN+ZH for now

> **English** | [中文](./0010-i18n.md)

- Status: Accepted
- Date: 2026-05-01

## Context

Until now every user-facing string in knowlet (CLI prompts, panel titles, web buttons, profile templates) was scattered through the code, with mixed defaults: English banners, a Chinese profile template, English error messages.

As knowlet approaches its first external users (post scratch-your-own-itch), the implicit "developer's mother tongue decides UI language" pattern is no longer sustainable:

- An English-native user creating a vault would see English banners but a Chinese profile template — jarring.
- Even Chinese users, working with English-speaking collaborators on shared vault configs / tasks / drafts, would encounter surprise mixed-language strings.
- Documentation already follows the [feedback_bilingual_docs](../../) bilingual rule; the code layer hadn't caught up.

The user stated on 2026-05-01: **"knowlet needs multi-language support, English default. For now, support English and Chinese."**

We need i18n at the code layer, with **English** as the explicit OSS default.

## Decision

### 1. Default language: **English**

- New vaults default to `config.general.language = "en"`.
- The default profile template defaults to English (`DEFAULT_PROFILE_TEMPLATE_EN`); Chinese is the alternate.
- Every UI string has an English version first; other languages translate.
- Docs (README, ADRs, design docs) continue to follow [feedback_bilingual_docs](../../) — Chinese-first or English-first drafting is fine, both versions in by commit time.

### 2. Implementation: **hand-rolled minimal catalog, no gettext / babel**

- `knowlet/core/i18n.py` — single file with two dicts (`_EN`, `_ZH`).
- Public API: `set_language(lang)`, `current_language()`, `init_from_env()`, `t(key, **vars)`, plus `all_keys(lang)` for the web layer to fetch a catalog.
- Missing key falls back to English; missing in English too **returns the key itself**, making missed translations visible during development.
- Current scale: ~80 keys. The gettext toolchain (`xgettext` / `.po` / `msgfmt`) overhead is not worth it. We can graduate later (>500 keys) with manageable migration cost.

**Why not gettext:** introduces a build step (.po → .mo); harder for an agent to add/remove keys quickly; `pybabel extract` doesn't reliably handle f-strings / Rich markup. A hand-written dict adds zero build dependency and any agent can edit the catalog directly.

### 3. Language source at runtime: **`config.general.language`, with KNOWLET_LANG override**

- Primary source: `[general] language = "en"` in `<vault>/.knowlet/config.toml` (new section).
- Override: env var `KNOWLET_LANG` for dev / test scenarios; not relied on for daily use.
- **Not** auto-detected from `Accept-Language` / locale — knowlet is a single-user local tool; auto-detection would create "why did the language change?" surprises.
- CLI calls `set_language(cfg.general.language)` via `_load_config_or_default` on every command.
- The web FastAPI lifespan calls it at startup; `/api/health` returns `language` so the frontend can display it; the frontend fetches `/api/i18n/{lang}` on bootstrap, populates `window.I18N`, then iterates `[data-i18n]` attributes to replace textContent / placeholder.

### 4. **Chat content language** (LLM replies) is **decoupled** from UI language

- The chat system prompt keeps "Reply in the same language the user used" — the LLM mirrors the user's input language, **not** `config.general.language`.
- Meaning: an English-UI user who types Chinese gets a Chinese reply. That's a feature (the bilingual-user real case), not a bug.
- Profile content injected into the system prompt is not translated — whatever the user wrote, the model reads.
- The mining-extractor prompt likewise: "write in the source's main language" — English source → English draft.

### 5. **CLI surface strings**: bulk-replace `print(...)` → `console.print(t("..."))`

- High-visibility paths first: vault init banner, config init wizard panels, chat REPL banner, slash help body, `_ensure_ready_or_wizard` onboarding copy.
- Low-visibility error messages ("task not found:" etc.) are **not** migrated all at once — translate as users actually hit them. Deliberate scope control: avoid "burning a day for 100% coverage" without a corresponding value increase.
- Migration shape: `t(key, **kwargs)`. Placeholders like `{root}` / `{model}` are filled via `str.format`.

### 6. **Web surface strings**: `data-i18n` attributes + JS apply

- HTML annotates static text with `data-i18n="key"` (sets textContent) or `data-i18n-placeholder="key"` (sets input placeholder).
- Dynamic toasts / status text use `tt(key)` in JS (lookup in `window.I18N` populated from `/api/i18n/{lang}`).
- Load order: health → read `language` → `loadI18n(lang)` → `applyI18n()` swaps strings → render data.

### 7. **Chat / sediment / mining-extractor prompts do NOT enter the catalog**

- These prompts are **LLM-facing**, not user-facing; `t()` is a user-layer mechanism and prompts shouldn't share it.
- Translating them adds no value — the UI language a user picked doesn't affect prompt-engineering effectiveness; double-translation (EN → ZH → LLM re-translates internally) only burns tokens.
- They remain English constants in `chat/prompts.py`.
- **Proxy compatibility is out of scope for this ADR**: some OpenAI-compat proxies (e.g. ones routing through Claude Code) don't strictly honor `role: "system"`. The mitigation (folding system instructions back into a user message) lives **next to the prompt that needs it** (see `chat/sediment.py:38-45`), not in the i18n layer. An earlier version of this section incorrectly implied "ADR-0008 already handled this" — that wasn't accurate; revised 2026-05-02.

## Consequences

### Benefits

- **OSS-friendly default**: external users land in English without needing Chinese.
- **Architecture pinned**: any new string goes into `core/i18n.py`, ensuring translation completeness and giving "miss visible at dev time" feedback (key returned literally).
- **No build step**: agents edit catalogs directly.
- **Backward-compatible by default**: old config.toml files without `[general]` get `"en"` automatically (Pydantic default); old vault profile files are **not** rewritten (`ensure_profile` only writes when missing).
- **Test-friendly**: `set_language("zh")` lets t() be asserted; `all_keys("zh")` vs `all_keys("en")` reveals coverage gaps.
- **No chat regression**: UI vs reply language are decoupled — bilingual users keep their happy path.

### Costs / constraints

- **First-pass catalog isn't 100% complete**. Low-visibility error messages stay as raw strings; intentional. Lint-able TODOs flag them.
- **HTML/JS need to keep catalog keys in sync**. Backend adds a key but frontend doesn't add `data-i18n` → fallback text shows. Watch for it in review.
- **Adding a new language**: one extra `_LANG_CODE` dict + one entry in `SUPPORTED_LANGUAGES`; ~50 lines of translation. Acceptable.
- **The profile template is not auto-regenerated on language switch**: an existing `users/me.md` stays in its creation language. This matches the "user owns vault content" principle — knowlet does not overwrite user files. Users can `rm users/me.md && knowlet user edit` to regenerate.
- **The chat REPL prompt itself swaps** between `[bold cyan]you[/bold cyan]` and `[bold cyan]你[/bold cyan]` based on i18n. Trivial; tests must not hardcode the literal string.

### Possible future extensions

- More languages (ja / ko / es / fr / de): ~50 lines per language.
- True RTL (ar / he): CSS logical properties. M5+.
- "Switch to English from now on" said in chat → already works (the system prompt tells the model to mirror user language); no new mechanism needed.
- A Web settings panel where the user clicks a language dropdown → reload `/api/i18n/{lang}` — five lines of JS, no ADR impact.

### Relation to existing ADRs

- **Follows** [ADR-0008](./0008-cli-parity-discipline.en.md): the catalog is a single backend source of truth; CLI / web / future desktop shells all read from it; no shell embeds its own strings.
- **Follows** [feedback_bilingual_docs](../../): docs stay bilingual; this ADR ships in both languages.
- **Does not conflict** with [ADR-0002](./0002-core-principles.en.md): language switching does not change data formats. Note / Card / Draft / Task frontmatter field names stay English (architecture convention); user-authored content remains user-owned.
- **Does not conflict** with [ADR-0005](./0005-llm-integration-strategy.en.md): LLM calls are unchanged; system prompts stay English.
