# 语音输入与 TTS 设计

> [English](./voice.en.md) | **中文**

> 活文档。**当前阶段一不优先实现语音功能**;本文保留作为后续阶段的设计参考。

## 当前定位

[ADR-0003](../decisions/0003-wedge-pivot-ai-memory-layer.md) 把破局点定为"AI 长期记忆层 + 减负型 PKM",目标用户以桌面端为主,移动端在阶段一仅做响应式 PWA。语音功能在阶段一不进入核心特性。

但语音在中长期仍有价值,尤其在场景 C(结构化重复记忆)的复习子流程中:通勤 / 走路 / 健身等不便看屏幕的场景。本文记录设计思路供阶段二/三参考。

## 两类核心场景

### 1. 语音答题(用户说话,系统理解)

复习场景:系统出题后,用户口头作答而非打字。

设计要点:

- **允许口语化与不精确表达**
- **LLM 做语义打分,而非字符匹配**
- **识别控制指令**:"我不会"、"跳过"、"再说一遍"
- **支持文字补录**:语音识别失败时允许用户追加打字修正

### 2. 语音出题(系统朗读,用户听)

通勤 / 走路 / 健身场景:

- **TTS 朗读卡片正面**,用户口头作答或心中作答后说"翻面"/"下一张"
- **节奏可调**:朗读速度、间隔时长适配场景
- **耳机控制**:配合蓝牙耳机控制按钮

## 已知坑位

### ASR 准确率

中英混杂术语(常见于技术学习场景)的 ASR 识别准确率显著低于纯中文 / 纯英文。

**对策:**

- 允许用户**补录文字**修正(交互必须低成本)
- 维护**用户专属术语词典**(从历史笔记 / Card 中提取),作为 ASR 后处理的纠错词表
- 关键复习场景下,优先用文字 + 语音双轨

### 语义打分的边界

LLM 语义打分会面临:答案部分对、超纲正确、误判申诉等情况。

**对策:**

- 评分粒度采用 "对 / 部分对 / 错 / 空答 / 跳过" 五档,不做 0–100 数值
- 每次打分**显示 AI 的判断理由**,允许用户一键覆盖
- 用户覆盖即反馈,记录后用于改进打分提示词

## 与 [ADR-0004](../decisions/0004-ai-compose-code-execute.md) 的衔接

语音输入的处理路径自然契合"AI 编排 + 原子执行":

- ASR / TTS 是原子能力(`transcribe_audio` / `synthesize_speech`)
- 语义打分由 LLM 编排,调用 `get_card / submit_review` 等已有 tool
- 不需要为语音单独搭一套工作流

## 何时进入路线图

语音特性进入路线图的触发条件(对应阶段二/三):

- 用户在场景 C 的复习量足够大,碎片场景需求明确
- ASR / TTS 模型成本与精度达到可接受阈值
- 移动端原生能力提供基础(麦克风 / 后台播放 / 锁屏控制)
