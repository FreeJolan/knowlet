# 语音输入与 TTS 设计

> 活文档。

## 为什么语音被低估

学习型 PKM 在桌面端被反复探索,但在**移动端碎片场景**(通勤、走路、做饭)往往只能"看"或"刷",学习行为基本停摆。语音输入与 TTS 输出能直接打开这个金矿场景,是学习型破局点(见 [ADR-0001](../decisions/0001-wedge-learning-first.md))之外少有的体验杠杆。

## 两类核心场景

### 1. 语音答题(用户说话,系统理解)

适用于复习场景:系统出题后,用户口头作答而非打字。

设计要点:

- **允许口语化与不精确表达**:不要求用户字面命中标准答案
- **LLM 做语义打分,而非字符匹配**:即便表述差异较大,只要语义命中要点即判对
- **识别控制指令**:"我不会"、"跳过"、"再说一遍"、"标记为难题"等需要被语音流识别
- **支持文字补录**:语音识别失败或专业术语识别错时,允许用户**追加打字修正**

### 2. 语音出题(系统朗读,用户听)

适用于通勤、走路、健身等不便看屏幕的场景。

设计要点:

- **TTS 朗读卡片正面**,用户口头作答或心中作答后说"翻面"/"下一张"
- **节奏可调**:朗读速度、间隔时长适配场景(走路 vs 跑步 vs 静止)
- **耳机优先**:配合蓝牙耳机控制按钮(单击翻面、双击下一张)

## 已知坑位

### ASR 准确率坑

中英混杂术语(常见于技术学习场景)的 ASR 识别准确率显著低于纯中文/纯英文。例如:

> "解释一下 BERT 里的 self-attention 是怎么工作的"

很多通用 ASR 模型会把 "BERT" 识别成"伯特"、"self-attention" 识别成各种音近词。

**对策:**

- 允许用户**补录文字**修正(交互必须低成本,例如"按住编辑")
- 维护**用户专属术语词典**(从其历史笔记中提取),作为 ASR 后处理的纠错词表
- 关键复习场景下,优先用文字 + 语音双轨,语音是加速器而非唯一通道

### 语义打分的边界

LLM 语义打分会面临:

- 用户答案部分对、部分错时如何给分?
- 用户答得"超纲正确"(讲到了卡片正面没要求的内容)如何处理?
- 误判时用户如何申诉?

**对策:**

- 评分粒度采用 "对 / 部分对 / 错 / 空答 / 跳过" 五档,不做 0-100 数值
- 每次打分**显示 AI 的判断理由**,允许用户一键覆盖
- 用户覆盖也是有用反馈,记录后用于改进打分提示词

## 与其他模块的协作

- **OCR**:扫书生成 Card → TTS 朗读 → 语音作答 → SRS 状态更新,形成纯免屏闭环
- **错题**:语音答错时,错因诊断需要识别"是真的不会"还是"识别错了"两种情况
- **领域实体**:语音答题数据是 Card.review_log 的一种来源,与文字答题平权
